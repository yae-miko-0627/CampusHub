# CampusHub 需求规格说明书

* 注：本文档基于提供的“00—大作业总览”交予AI，按照IEEE830标准生成，后续由团队成员进行人工修改完善，若未标注任何标签，则说明直接移用自“00—大作业总览”

## 1. 引言

### 1.1 编写目的

本文档旨在详细描述 CampusHub 平台的业务需求、功能需求和非功能需求，为后续的架构设计、详细设计及编码提供基准依据。

### 1.2 项目背景

大学校园内存在大量碎片化的互助需求（如快递代取、学习辅导等），目前缺乏统一且具信任机制的平台。CampusHub 旨在通过一个集中的 Web 平台，提高信息匹配效率并建立校园互助信任体系。

### 1.3 范围

本项目涵盖用户管理、需求发布、智能推荐、订单流程、评价信用及管理后台等模块。

### 1.4 术语与缩写

- CampusHub：校园互助服务平台。
- 需求方：发布互助请求的用户。
- 服务方：提供互助服务的用户。
- P0/P1/P2：表示功能优先级或阶段划分。

### 1.5 参考资料

- IEEE 830 软件需求规格说明书推荐规范。
- 《软件工程与计算 II》课程相关要求与阶段文档。

## 2. 总体描述

### 2.1 产品前景

CampusHub 作为“软件工程与计算 II”课程的大作业，采用前后端分离架构，并在开发全生命周期中引入 AI 协同模式。

### 2.2 产品功能

#### 2.2.1 用户管理 [AI生成]

支持基于学号的身份验证及资料维护。

#### 2.2.2 互助市场 [AI生成 + 人工修改]

* 注：补充了互助需求的分类（快递代取、学习辅导、二手交易、资料分享、失物招领等）

提供需求发布、多维度筛选（分类、时间、距离）及详情查看。
主要负责交易类互助需求，如快递代取、学习辅导、二手交易等，后续可扩展至其他类型的互助需求。

#### 2.2.3 交流论坛 [人工补充]

提供学习，生活等需求的交流讨论区，支持发帖、评论及点赞功能。
主要负责非交易类互助需求，如学习交流、资料分享、失物招领等，后续可根据实际需求调整或增加其他类型的交流讨论区。

#### 2.2.4 交易闭环 [AI生成]

涵盖从接单、进行中到确认完成的完整状态机管理。

#### 2.2.5 信用体系 [AI生成]

通过双向评分机制建立用户画像。

#### 2.2.6 管理后台 [AI生成]

支持内容审核、用户管理与基础数据统计。

### 2.3 用户特征

- 需求方：发布互助请求（如快递代取、咨询、二手交易等）的学生。
- 服务方：提供服务并赚取报酬或交换条件的学生。
- 管理员：负责内容审核、用户管理及数据统计看板的运营人员。

注：同一学生用户可兼具需求方和服务方双重身份。

### 2.4 运行环境

- 后端：Spring Boot 3、JDK 17、Maven。
- 前端：Vue 3、Node.js 20.x。
- 数据库：MySQL。
- 开发工具：Git、IDEA、浏览器及 AI 编程工具。

### 2.5 设计与实现约束

- 周期：10 周开发时长。
- 工作量：团队总计约 240 人时。
- 关键技术：建议前后端分离，鼓励使用 AI 编程工具辅助。

### 2.6 假设与依赖

- 用户均为校园内学生或教师等校内成员。
- 系统依赖学号信息完成身份校验。
- 校园场景中用户规模相对有限，便于分阶段迭代。

## 3. 具体需求

### 3.1 功能需求

#### 3.1.1 模块一：用户管理（P0）[AI生成 + 人工修改]

* 注：修改注册/登录部分，不支持第三方注册登录，改为仅支持学号+密码的方式

- 注册/登录：支持学号+密码登录。
- 个人中心：管理头像、昵称、学院、联系方式及查看历史订单。

#### 3.1.2 模块二：需求发布与浏览（P0）

- 发布需求：填写标题、描述、分类（快递代取、学习辅导、二手交易、资料分享、失物招领等）、地点、时间、报酬。
- 多维检索：支持按分类筛选，并支持时间及距离排序。
- 需求详情：展示需求内容、发布者信息、状态与交互入口。

#### 3.1.3 模块三：订单流程管理（P0）

- 状态转换：接单确认 → 进行中 → 完成 → 评价。
- 确认机制：需需求方与服务方双方确认方可结算。
- 过程记录：保留关键状态变更记录以便追踪。

#### 3.1.4 模块四：评价与信用体系（P1）

- 双向评价：订单完成后支持 1-5 星评分及文字评语。
- 信用模型：基于评分计算信用分并在个人主页展示信用等级。
- 信用展示：支持根据历史评价形成基础信用画像。

#### 3.1.5 模块五：消息通知（P1/P2）

- 系统通知：站内推送接单、状态变更等消息。
- 即时通讯（P2）：支持需求方与服务方的一对一简易文字交流。

#### 3.1.6 模块六：管理后台（P1）

- 内容审核：审核违规内容、广告信息及不当言论。
- 用户管理：处理用户封禁、恢复与基础信息查看。
- 数据统计：查看基础运营数据与内容分布情况。

### 3.2 非功能需求

#### 3.2.1 性能需求

核心页面（首页、列表页）加载时间应小于 2 秒。

#### 3.2.2 安全性需求

- 密码必须加密存储。
- 系统需具备基础的 SQL 注入和 XSS 攻击防护能力。
- 关键操作需具备基本的权限控制。

#### 3.2.3 易用性需求

界面简洁，完成发布或接单等核心路径的操作步数不超过 3 步。

#### 3.2.4 可维护性需求

代码结构需清晰，核心业务逻辑需提供单元测试覆盖。

#### 3.2.5 可扩展性需求

系统应支持后续增加新地互助分类、评价维度与消息类型。

## 4. 最小可行产品（MVP）

为确保 10 周内交付，项目必须优先完成以下 P0 级功能：

- 用户注册、登录及个人资料。
- 需求的发布、浏览、详情查看与分类筛选。
- 完整的接单及完成流程。

## 5. 验收标准

### 5.1 交付物

需包含需求文档、架构文档、源代码及测试报告。

### 5.2 功能验收

- 用户能够完成注册、登录与个人资料修改。
- 用户能够发布、浏览、筛选并查看需求详情。
- 用户能够完成接单、进行中、完成与评价的完整流程。
- 订单完成后可进行双向评价并查看信用信息。

### 5.3 过程验收

必须按阶段提交《AI 协作反思日志》，记录 Prompt 策略及 AI 纠错过程。
